Navigation, Localization, and Mapping

    Navigation is one of the most essential functionalities for robots. Robots use the given map to get the location and then find the direction to reach the target. However, The given map is an assumption. Most of the time, there is no map or the map is not accurate as it is produced. Therefore, to build the map of the unknown/known environment becomes an indispensable problem in robotics.
    While to do localization and mapping at the same time is a chicken-and-egg problem, there are several well-known approximation algorithm to solve it such as particle filter, extended Kalman filter ... etc. In my work, I have tried particle filter to do Simultaneous Localization and Mapping (SLAM).

Mobile Robot Navigation System

    Course: Robotics, Department of Computer Science, National Taiwan University

This is my first robotics project. I worked with one classmate to control a pioneer-like mobile robot to autonomously navigate in dynamic scenes with a Kinect sensor to reach a ball. I constructed the system to combine the mobility with the perception's estimation. In addition, I design a simple obstacle avoidance algorithm and grid-based path planning.

Localization and SLAM Competition

    Course: Robot Perception and Learning, Department of Computer Science, National Taiwan University

We use an Xtion sensor and a manually controlled mobile robot to get RGB, depth, and odometry information in both competitions. My main strategy to accomplish these two tasks is via Monte Carlo method (particle filter). As you can see in the video, red dots are randomly generated particles based on the given mean and sigma. The challenge in the localization task is that the robot will pass by some dark places where RGB information is helpless. In addition, there are some walking students whom may cause inaccurate depth alignment. On the other hand, in the SLAM task, we have to do loop detection and try to handle loop closure problem.

The localization competition is an online demonstration. The shown video is based on the recorded images and odometry measurement. There are four main steps in the localization process.
  1. Predict the possible position distribution by using the odometry readings.
  2. Weight the particles based on the distribution and correct some particles by using depth information. (Here, due to real-time performance issue, I only use one row of the depth image. It can be treated as a kind of 2D scan.)
  3. Estimate the most likely position after computing the correlation of the current RGB image and integrating with the previous weighting result. (To find the correlation between images, I use SIFT feature detector to match and compute weighting, #matched_keypoints/#all_keypoints.)
  4. Resample particles.


In the SLAM videos, the measurements are captured at our RoboCup field. To build the map, I exploit the 2D scan to do Iterative Closest Point (ICP) alignment. Then, do the above localization and use the estimated position to refine self position. The difference between the left and right video is that the right video has a map refinement process after loop detection. Since we take the video at a RoboCup field, one of the most obvious alignment differences are the goal and goal zone. As you can see, in the right video the goal's yellow pillar and lines of goal zone have much fewer miss aligned duplicates than the left video which means it builds a better map based on the estimated position.

WiFi-SLAM

The massive deployment of wireless network in the indoor environment paves a way not only for doing localization, but also helping build a wireless map. Without knowing the location of the wireless landmarks, SLAM provides a solution to build the map and do localization at the same time. Based on the paper, Efficient Probabilistic Range-Only SLAM, I build the WiFi-SLAM system with a Raspberry PI and iRobot vacuum cleaner and test in an office. Since the WiFi signal is noisy and unstable when the robot is moving, I have implemented a bucket filter to stabilize the signal strength. (In the experiments, I have set up multiple Raspberry PI which can both receive WiFi signal and be a hot-spot.)
Compare with the original odometer reading, it shows excellent performance in practice in some cases. However, I haven't figured out how to deal with symmetric problem when changing the orientation. I give an unsuccessful result to address this intrinsic problem when estimating orientation.
